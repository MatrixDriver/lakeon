Section 03

为什么能这么快

核心论点：不是"AI 替代程序员"，而是"专家 + AI = 超级团队"

效率三层叠加模型

I
专家经验 — 方向正确
架构师长期思考 + 3 位专家讨论，开工前已有清晰架构蓝图
II
AI 执行力 — 速度极快
设计确认后，全栈编码 + 测试 + 部署脚本同时生成
III
方法论 — 减少浪费
三段式流程、CLI E2E、Skill 沉淀，返工率压到 27%
关键推论：AI 是能力放大器，放大的是使用者已有的判断力和经验。

AI 帮助创新：从分析到实现

创新流程：
1. 分析业界
AI 分析大量材料
2. 头脑风暴
探索多种路径
3. 人决策
根据竞争力需要
4. 快速实现
AI 加速落地
案例：热启动从 KPI 目标 <10s 优化到 3ms，提前达成 2026 年目标

AI 工具组合使用

主力工具：Claude Code (Opus 4.6) - $100/月订阅，97% cache read 效率
辅助服务：
  • 硅基流动 - Embedding + LLM API
  • OpenRouter - Embedding + LLM API
  • 国内 LLM 中转服务
  • 自动分析竞品：火山云、阿里云、AWS 对应云服务

实践一：人定目标，AI 结构化路线图▶

核心理念：你有产品愿景，AI 帮你拆解成可执行的阶段化计划。人负责"做什么"和"为什么"，AI 负责"怎么做"和"分几步"。

ROADMAP 演进过程

  • 初版 (03-04)：Stage 0-5（本地 → OBS → RDS → CCE → 控制台 → SRE）
  • 第一次扩展 (03-06)：加入 Stage 6b（PG 导入）
  • 第二次扩展 (03-08)：加入 Stage 10a-10e（备份、分支、审计、权限）
  • 第三次扩展 (03-17)：加入 Stage 11-12（时间旅行、弹性节点池）
  • 最终版 (03-22)：16 个阶段，覆盖全功能

关键：路线图不是一次定好的，是随着项目推进、技术验证、需求变化不断调整的。

实践二：Design Spec → Plan → Code 三段式▶

核心理念：AI 写代码很快但容易偏离方向，前期 review 文档比后期修代码成本低 10 倍。

正面案例：Stage 15b 知识库增强

设计文档 (30min) → 实现计划 (30min) → 编码 (7h) → 一次性通过，无返工

1 天完成 4 个特性：文档标签、查询重写、重排序、表知识库

反面案例：Stage 15 KB Pipeline

没有仔细审查设计就开始编码 → AI 方案用独立 kb-write Pod 写入 → 触发 Neon WAL 冲突 → 反复调试浪费大量时间

教训：如果在设计阶段就讨论"多个 Pod 同时写入 Neon 会怎样"，30 分钟就能发现并修正。

Fix 提交占 27% (124/466)，很多是设计不清晰导致的返工。严格 review 设计可降到 15% 以下。

实践三：多窗口并行 + worktree 隔离▶

核心理念：AI 可以同时处理多个任务，人的瓶颈是注意力切换。

窗口用途示例
A主特性开发Lakeon Stage 16 数据湖
B同项目另一特性 (worktree)Stage 15c 切片管理
C另一个项目MemOS 记忆库集成
D调试/部署/运维CCE 环境问题排查

避免打架的关键

  • 不同窗口不要改同一个文件
  • worktree 做特性开发，主目录做集成和部署
  • 每个窗口有独立的 Memory context

理论上 4 个窗口 = 4x 效率，实际 ≈ 2-3x（有等待和协调成本），但比单窗口快得多。

实践四：CLI 驱动的 E2E 测试闭环▶

核心理念：让 AI 能自己验证自己的工作，形成自动化调试循环。

传统方式 vs CLI 驱动方式

传统：AI 写代码 → 人手动测试 → 发现问题 → 告诉 AI → AI 修 → 人再测试

CLI 驱动：AI 写代码 → AI 写测试 → AI 跑测试 → 失败 → AI 读错误 → AI 自己修 → 再跑 → 通过

关键差异：人从测试执行者变成验收者，AI 完成了整个调试循环。

真实案例

测试数据库创建后立即连接失败 → AI 自动判断 compute pod 未就绪 → 添加等待逻辑 → 重新跑测试通过。耗时 3 分钟（人工至少 15 分钟）。

实践五：Skill 沉淀 — 把经验变成 AI 的能力▶

核心理念：把踩坑经验固化为 AI 的行为规范，团队可共享复用。

Skill用途价值
huawei-cloud-style华为云 UI 风格参考真实截图提取规范，AI 生成一致风格前端
sre-consoleSRE 运维模板标准化运维页面结构
hcloud华为云 CLI封装 CLI 用法，避免 REST API 出错
deploy部署前预检防止凭据丢失、tag 错误等历史事故
weekly-report周报生成自动化格式化周报

案例：deploy Skill

手动部署出过多次事故（RDS 配置丢失、Helm 冲突、忘记测试），把所有事故总结成检查清单写成 Skill。AI 部署前自动调用：检查环境变量、强制跑测试、确认 tag、禁止手动 kubectl。部署事故从频繁发生降到 0。

实践六：用 /insight 让 AI 帮你改进工作方式▶

核心理念：AI 不只是写代码的工具，也是改进工作流程的顾问。

采纳的改进

  • 创建 Skill 自动化重复性工作 — AI 观察到反复手动检查部署环境变量，建议创建 deploy Skill，部署事故率降为 0
  • 调整会话策略 — AI 发现长会话里混杂太多不相关任务，建议一个会话一个主题，AI 输出质量明显提升
  • 优化 worktree 使用 — AI 建议用 worktree 隔离并行特性开发，多窗口并行效率从 1.5x 提升到 2-3x

这是一种"元认知"能力 — AI 帮你反思自己的工作方式，而不只是执行任务。

AI 开发规范：还需要补齐什么

当前只有 Superpowers + 项目级 Skill + Memory 系统，要让 AI 开发产出的代码达到产品上线标准，还需要建立以下规范：

1. AI 代码 review 标准

谁来 review AI 写的代码？用什么标准？
需要明确 AI 代码与人工代码的 review 流程差异。

2. 安全审计流程

AI 是否引入了安全漏洞？如何检测？
需要建立 AI 代码的安全扫描和审计机制。

3. 可维护性评估

AI 写的代码别人能看懂吗？注释够吗？
需要可维护性标准和文档化要求。

4. 性能基准

AI 写的代码性能如何？有没有明显的性能问题？
需要建立性能测试基准和回归机制。

5. 容灾测试

故障转移、数据恢复是否可靠？
AI 开发的系统需要经过完整的容灾演练。

6. CI/CD 集成

如何把 AI 开发流程集成到现有 CI/CD？
需要将 AI 质量门禁纳入流水线。

当前已有的： Superpowers 内置实践 5 个项目级 Skill Memory 跨会话上下文

结论：只用 Superpowers 远远不够。AI 开发规范需要覆盖代码 review、安全审计、可维护性、性能、容灾、CI/CD 六个维度，才能让 AI 产出的代码达到产品上线标准。
启示：AI 大幅加速了原型开发，但产品化需要团队协作，合规和流程规范也需要适应 AI 开发模式做出改变。
