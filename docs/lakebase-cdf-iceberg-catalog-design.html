Lakebase CDF · Lakeon-managed Iceberg · 方案讨论稿

用用户 Lakebase 承载 Iceberg Catalog 事务，用 OBS 承载开放表格式

本文整理 Lakeon 在 Lakebase Core 内实现 CDF 写 Lakeon-managed lakehouse table 的方案：用户业务写 Lakebase/Postgres，Lakeon 捕获变更并写成可由 Iceberg REST Catalog 访问的表。默认热路径以用户 Lakebase 中的 catalog、planning index 和事务状态为准，OBS 主要保存 Parquet data/delete files；标准 Iceberg metadata/manifest 可在用户需要兼容、恢复、审计或导出时按需物化。

实时性目标 MVP 10-30 秒；优化后 3-10 秒；不承诺 sub-second Iceberg。
客户端兼容 使用标准 Iceberg REST Catalog；用户不需要换 PyIceberg、Spark、DuckDB 客户端。
核心绑定点 绑定 Lakeon Catalog，不开放直接 OBS metadata 路径作为接口。

1. 设计目标

目标不是把 Lakebase 伪装成一个普通对象存储 Iceberg 表，而是构建一种 Lakebase-native 的近实时 lakehouse 能力： Lakebase 负责事务、权限、branch、低延迟 Catalog load 和 scan planning；OBS 负责 Parquet data/delete files、长期存储，以及按需物化后的标准 Iceberg metadata/manifest。

Lakebase CDF Iceberg REST Catalog 用户 Lakebase 元数据 OBS 标准文件 Lakebase branch dbay-agent Python 作业
关键决策：这些表定义为 Lakeon-managed lakehouse tables。用户通过标准 Iceberg 客户端连接 Lakeon REST Catalog 读写；直接读取 OBS 上的 metadata.json、manifest 或 data files 不作为受支持接口。默认不是“裸 Iceberg 表”，而是 Lakebase-backed catalog + planning index + OBS Parquet 的托管表。

2. 总体架构

Lakeon API 同时承担 Lakebase 控制面、CDF 管理、Iceberg REST Catalog 和 server-side scan planning。Catalog 的 authoritative row 与 planning index 存在用户 Lakebase 的系统 schema 中；OBS 默认保存 Parquet data/delete files，并在需要时物化标准 Iceberg metadata/manifest；dbay-agent 只通过稳定 HTTP/API 和标准 Iceberg Catalog 访问，不共享 Lakeon RDS 表。

3. CDF 到 Iceberg 的数据流

步骤 动作 事务边界 可见性
1 应用写入 Lakebase/Postgres 业务表。 用户 OLTP 事务。 立即对 Lakebase 查询可见。
2 用户打开 CDF 后，Lakeon 先执行 initial backfill，把源表当前快照写成第一批 Parquet 和 planning index。 捕获 backfill LSN，作为后续增量消费边界。 backfill snapshot commit 后，Iceberg 客户端可读到完整初始表。
3 CDF Worker 从 backfill LSN 之后通过 logical decoding / WAL 读取增量变更。 按 LSN 顺序消费，记录 batch 边界。 增量尚未对 Iceberg 可见。
4 Worker 将变更写成 Parquet data/delete files，并生成 Lakebase planning index 变更。 OBS 数据文件先写入未提交状态；manifest 默认不立即生成。 直接 OBS 路径不是支持接口。
5 Worker 调 Lakeon REST Catalog commit。 用户 Lakebase 中 CAS 更新 catalog row，同时记录 CDF offset。 CAS 成功后 Iceberg snapshot 对 catalog 客户端可见。
6 dbay-agent 或外部引擎 load table / scan。 load table 从用户 Lakebase 读取热 metadata；支持 server-side planning 的客户端从 Lakebase planning index 获取 FileScanTask，再读取 OBS Parquet。 看到 branch 当前 catalog pointer 指向的 Iceberg snapshot。
实时性建议：打开 CDF 后先完成 initial backfill，目标表才标记为可读；backfill 后增量 CDF 第一版目标 10-30 秒端到端延迟，优化后可压到 3-10 秒。

4. 中间元数据方案

最小 pointer 方案性能收益不够；全量复制 Iceberg metadata 又会制造双真相。推荐中间方案升级为：用户 Lakebase 存当前 metadata 的热副本、catalog 事务状态和 scan planning index；OBS 默认保存 Parquet data/delete files，标准 Iceberg metadata/manifest 作为按需物化副本。

位置 保存内容 是否 authoritative 原因
用户 Lakebase _lakeon_iceberg.tables current_metadata_location、current_metadata_json、metadata_hash、commit_lsn、branch scope、export 状态。 当前可见版本和 Catalog 事务以这里为准。 低延迟 load table、CAS commit、权限、审计、branch 隔离。
用户 Lakebase planning index snapshots、data_files、delete_files、partition values、column bounds、record count、file size。 Lakeon 默认 scan planning 以这里为准。 避免客户端热路径读取 OBS manifest；用 Lakebase 索引做文件裁剪和权限控制。
OBS Parquet data/delete files 实际数据文件和删除文件。 真实数据内容以这里为准。 保持列式分析存储、对象存储成本和跨引擎扫描基础。
OBS lazy Iceberg metadata / manifest 按需生成的 metadata/*.metadata.json、manifest-list、manifest。 标准兼容、恢复、审计和导出副本。 只有用户需要外部 Iceberg 生态或恢复审计时物化，避免每次 commit 增加 OBS 小文件和延迟。
Lakeon API 内存缓存 table pointer、metadata hash、权限结果、短 TTL table summary。 非 authoritative。 降低 REST Catalog 热路径延迟。 
-- 用户 Lakebase 中的核心 catalog row，示意
UPDATE _lakeon_iceberg.tables
SET current_metadata_location = :new_metadata_location,
    current_metadata_json = :new_metadata_json,
    metadata_hash = :new_metadata_hash,
    current_snapshot_id = :new_snapshot_id,
    metadata_version = metadata_version + 1,
    last_commit_lsn = :end_lsn,
    export_status = 'NOT_MATERIALIZED',
    updated_at = now()
WHERE table_id = :table_id
  AND branch_id = :branch_id
  AND current_metadata_location = :expected_old_metadata_location;

5. Lazy Iceberg Manifest Materialization

默认热路径不要求每次 commit 都写 OBS 上的 Iceberg metadata/manifest。Lakeon 可以先把 commit 记录、snapshot、data file index 写入用户 Lakebase，并把 Parquet data/delete files 写入 OBS；当用户需要标准 Iceberg 兼容、审计导出、外部引擎直读或灾难恢复时，再把 Lakebase planning index 物化成标准 Iceberg metadata 和 manifest。

模式 默认访问 OBS manifest 状态 适用场景
managed_only 只支持 Lakeon REST Catalog；scan planning 由 Lakebase planning index 完成。 不保证每个 snapshot 立即有标准 manifest。 默认模式，最低 commit 延迟，绑定 Lakeon 增强能力。
iceberg_exported Lakeon REST Catalog 仍是主入口；也可以交付标准 Iceberg metadata/manifest 给外部生态。 后台物化标准 metadata.json、manifest-list、manifest。 外部 Spark/Trino/Databricks 接入、审计、离线恢复、客户交付。
取舍：lazy manifest 会让默认表不再是“随时可绕过 Catalog 直读的裸 Iceberg 表”。这是有意的产品边界。Lakeon-managed 表必须通过 Lakeon REST Catalog 获得事务、权限、branch、planning index 和 GC 能力。

6. Query Path Guidance，不是第一期自动路由

第一版不做透明的 automatic query path selection。Iceberg 客户端通过 Lakeon REST Catalog 读 Lakeon-managed table，最终仍然扫描 OBS 上的 Parquet data files；PostgreSQL 客户端继续读 Lakebase 原表。PG 客户端只有在未来启用 Lakebase serving projection，或直接查询源表时，才能得到 Lakebase row-store 的低延迟好处。

客户端 / 场景 第一期路径 能得到的好处 不能承诺的能力
Iceberg 客户端 / 分析扫描 Lakeon REST Catalog + OBS Parquet。 Catalog load、权限、branch、CDF snapshot、lazy export；支持 REST scan planning 的客户端还可减少 manifest planning 时延。 不能保证所有标准 Iceberg 客户端都不读 manifest；也不能避免实际 Parquet 数据扫描。
PG 客户端 / 点查 Lakebase 原表。 普通 PostgreSQL driver 即可获得 Lakebase 低延迟索引访问。 PG 客户端不会自动查询 Iceberg Parquet；Lakeon 也不会在第一期伪装成 PG 查询引擎。
PG 客户端 / 未来服务化结果 未来 Lakebase serving projection。 模型特征、画像、风控分等 current-state 结果可用普通 PG 查询。 这是独立大特性，需要投影表、幂等 apply、schema evolution、delete/tombstone 和延迟 SLA。
如果要让 PG 或 SQL-over-HTTP 客户端直接查询 Iceberg 表，Lakeon 必须内置或集成一个面向 Parquet/OBS 的查询引擎，并实现成本控制、并发、缓存、spill 和 planner。这不属于第一期 CDF/Catalog 计划，应放到未来阶段。 
-- 查询路径示意
SELECT * FROM orders_current WHERE order_id = 'o_123'
-- -> 第一版：Lakebase source OLTP table
-- -> 未来：Lakebase serving projection

SELECT region, count(*), sum(amount)
FROM orders_cdf
GROUP BY region
-- -> Iceberg engine -> Lakeon REST Catalog -> OBS Parquet

7. Branch 模型

用户只感知 Lakebase branch。Iceberg branch/tag 作为标准 metadata 能力保留，但不作为第一版产品主心智暴露。

用户看到的 branch

  • database = app_prod
  • branch = main / experiment / replay
  • 每个 branch 下有自己的 Iceberg catalog pointer。
  • dbay-agent 按 branch 连接 Lakeon REST Catalog。

系统内部的 branch

  • Lakebase branch 隔离 _lakeon_iceberg.* catalog rows。
  • OBS metadata/data 文件按引用共享，不复制全量文件。
  • Iceberg refs 保持标准格式，用于兼容高级引擎语义。
  • GC 需要综合所有 Lakebase branches 和 Iceberg snapshots。
产品表达：不是“Lakebase branch + Iceberg branch 两套概念”，而是“Lakebase branch 下的 Iceberg 表自动跟随分支”。Iceberg refs 只有在用户显式使用高级 Iceberg 能力时才出现。

8. dbay-agent 如何访问

dbay-agent 不直连 Lakeon 元数据库，也不共享 JPA 表。它通过 Lakeon API 获取 catalog 配置，并使用标准 Python / SQL 引擎访问 Iceberg 表。 

# PyIceberg 示例，具体参数由 Lakeon API 下发
from pyiceberg.catalog import load_catalog

catalog = load_catalog(
    "dbay",
    uri="https://api.dbay.cloud:8443/api/v1/iceberg/catalog/app_prod/main",
    warehouse="obs://dbay-mainstore/iceberg/tenant/app_prod",
    token="service-or-user-token",
)

table = catalog.load_table("public.orders_cdf")
scan = table.scan().to_arrow()

对 DuckDB、Spark、Ray 作业也遵循同一原则：客户端保持标准，Catalog URI 指向 Lakeon，OBS 凭证由 Lakeon REST Catalog 按需下发或签名。

9. 优点

优点 说明
低延迟 Catalog load 标准 REST Catalog 响应可以直接返回 metadata JSON；Lakeon 从用户 Lakebase 读取热副本，避免每次从 OBS 拉 metadata JSON。
低延迟 scan planning 支持 REST server-side scan planning 的客户端可以让 Lakeon 用 Lakebase planning index 返回 FileScanTask，避免客户端读取 OBS manifest。
事务能力强 Iceberg commit requirements 映射到用户 Lakebase/Postgres CAS 和事务，CDF offset 与 Iceberg snapshot 可原子绑定。
branch 能力自然 Lakebase branch 隔离 catalog pointer 和热 metadata，用户只理解 Lakebase branch，不需要学习两套 branch。
客户端生态兼容 不 fork Iceberg 客户端，不改 Iceberg 文件格式；PyIceberg、Spark、DuckDB 等标准客户端只需配置 Lakeon REST Catalog。
成本归属清楚 Catalog 热元数据和 planning index 占用户 Lakebase 容量；Parquet data/delete files 占 OBS；标准 metadata/manifest 可按需物化；Lakeon RDS 只保存最小控制面索引。
查询路径边界清楚 Iceberg 客户端走 Catalog + Parquet；PG 客户端走 Lakebase 原表，未来可走 serving projection。第一期不承诺透明自动路由。
形成产品绑定 增强能力在 Lakeon Catalog 层：权限、审计、branch、GC、compaction、CDF offset。用户绑定的是服务能力，不是客户端私有协议。

10. 缺点与应对

问题 影响 应对
默认不是裸 Iceberg 表 如果不立即物化 manifest，用户不能绕过 Lakeon Catalog 直接读 OBS 表。 产品定义为 Lakeon-managed table；需要外部兼容时进入 iceberg_exported 模式。
元数据重复更多 Lakebase 中除了当前 metadata，还会保存 data_files / stats planning index。 这是换取低延迟 planning 的核心成本；用分区、压缩、retention 和 GC 控制膨胀。
commit 变重 每次 CDF commit 需要写 Parquet，并更新 Lakebase planning index。 按 micro-batch 提交；data file index 批量 upsert；小文件由 compaction 后台处理。
Catalog compute 冷启动 用户 Lakebase compute 休眠时，commit/load 可能需要唤醒。 写入作业期间 keep-warm；读路径使用 Lakeon 短 TTL cache；生产 SLA 按 warm/cold 分开展示。
GC 复杂 需要考虑所有 Lakebase branches、Iceberg snapshots、未完成 commit 和 active query lease。 GC 是 Lakeon 内部职责，不暴露给用户；后台先 mark reachable，再延迟删除 OBS 文件。
必须收口所有写入 如果外部引擎绕过 Lakeon Catalog 直接写 OBS，Lakebase catalog 和 planning index 会失效。 产品定义为 Lakeon-managed Iceberg tables；OBS 路径不是支持接口；凭证和权限只通过 Lakeon Catalog 下发。
Serving table 是额外副本 如果为分析产物或跨表 current-state 维护 Lakebase serving table，会增加 row-store 存储和 apply 工作。 放到未来 PG Serving 阶段；只对用户声明的低延迟查询或物化结果启用；源 OLTP 表本身可直接承担点查。
PG 查询 Iceberg 需要查询引擎 普通 PG driver 不能直接扫描 OBS Parquet，也不会自动使用 Iceberg Catalog。 如果产品需要这个体验，规划为 Lakeon Query Engine 阶段，不纳入第一期 CDF/Catalog。
不是开放裸表模式 用户不能把 OBS path 当成完整产品接口。 这是有意的竞争力边界：开放客户端协议，绑定 Catalog 服务能力。

11. 推荐分阶段

详细路线图见 docs/lakebase-cdf-iceberg-roadmap.md。下面只保留用于评审的阶段摘要。

阶段 交付 成功标准
MVP Lakeon REST Catalog、用户 Lakebase catalog tables、initial backfill、CDF append changelog、OBS Parquet data files、基础 PyIceberg load。 打开 CDF 后先生成完整初始 snapshot；backfill 后 10-30 秒内从 Lakebase 增量写入到 Lakeon-managed table 可见。
性能版 metadata 热缓存、Lakebase planning index、REST server-side scan planning、writer keep-warm、并发表 writer、小文件 compaction、Console 状态页。 常规负载 3-10 秒可见；Catalog load 和 scan planning 不依赖客户端读取 OBS metadata/manifest。
Lazy Export 版 按需物化标准 Iceberg metadata/manifest，支持 managed_only 和 iceberg_exported 两种状态。 默认 commit 不写 manifest；用户需要兼容、审计、恢复时可生成标准 Iceberg 文件。
Branch 版 Lakebase branch 下的 catalog pointer 和 planning index 自动隔离；branch-aware GC。 同一张 Lakeon-managed table 在不同 Lakebase branch 可独立提交和读取。
PG Serving 版 为声明的 current-state / gold 结果维护 Lakebase serving projection。 PG 客户端用普通 PostgreSQL driver 点查低延迟结果；这不是自动查询路由。
Lakeon Query Engine 版 内置或集成 Iceberg/Parquet 查询引擎，提供 SQL API 或 PG-compatible gateway。 Lakeon 才能真正自动选择 Lakebase source、serving projection 或 OBS Parquet 执行；这是未来大阶段。

12. 需要进一步确认的问题

  • 第一版 CDF 表是只做 append changelog，还是同时维护 current-state planning index。
  • Initial backfill 对大表的分页策略、进度展示、失败重试和 backfill LSN 捕获方式。
  • OBS 凭证下发方式：REST Catalog vended credentials、短期 STS，还是 Lakeon 内部代理写入。
  • 是否第一版就支持 DELETE files 和 MERGE-on-read，还是先用 append changelog 简化。
  • Catalog 热 metadata JSON 与 planning index 在 Lakebase 中的压缩、分区、大小限制和迁移策略。
  • 哪些客户端/引擎优先验证 REST server-side scan planning。
  • lazy manifest 的触发条件：手动导出、外部引擎接入、审计快照、周期性 materialization，还是按表策略。
  • Lakebase serving table 的启用条件：只为分析产物，还是也为源表 current-state 生成独立副本。
  • GC 的最短保留期、active query lease、跨 branch reachability 算法。
