Section 03

为什么能这么快

核心论点：不是"AI 替代程序员"，而是"专家 + AI = 超级团队"

效率三层叠加模型

I
专家经验 — 方向正确
架构师长期思考 + 3 位专家讨论，开工前已有清晰架构蓝图
II
AI 执行力 — 速度极快
设计确认后，全栈编码 + 测试 + 部署脚本同时生成
III
方法论 — 减少浪费
三段式流程、CLI E2E、Skill 沉淀，返工率压到 27%
关键推论：AI 是能力放大器，放大的是使用者已有的判断力和经验。

AI 帮助创新：从分析到实现

创新流程：
1. 分析业界
AI 分析大量材料
2. 头脑风暴
探索多种路径
3. 人决策
根据竞争力需要
4. 快速实现
AI 加速落地
案例：热启动从 KPI 目标 <10s 优化到 3ms，提前达成 2026 年目标

具体方法

三段式开发

Design Spec → Plan → Code
确认设计后再编码，减少返工

多窗口并行

3-4 个 AI 窗口 + git worktree 隔离
并行推进多个 Stage

CLI 驱动测试闭环

AI 自己写测试 → 跑 → 修 → 通过
形成自动化质量闭环

Skill 沉淀

部署、前端风格、SRE 模板、云操作
封装为可复用 AI 技能

Section 04

局限性与风险

AI 做不到的事（必须人判断）

  • ✖ 云服务兼容性
    pg_search BM25 与 Neon SMGR 不兼容 → 数据库 crash
  • ✖ 网络拓扑
    CCE overlay 不通 RDS，CCI 不通 OBS
  • ✖ 跨境限制
    ISP 拦截 Railway → 中国 POST 请求
  • ✖ 容器限制
    CCE Autopilot setrlimit(CORE) 失败

需要关注的数据

27% 提交是修 bug (124/463)

快不等于对

47 次部署

开发-部署循环快，线上调试多

需要有经验的开发者驾驭 AI 输出质量
