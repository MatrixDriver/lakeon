## 1. 项目背景与数字概览

### 1.1 项目初始目标

构建支撑 Agent 的 **Serverless 数据底座**，以 Neon 为内核的 Lakebase 为知识库、记忆库、AI 多模数据湖提供统一支撑。

### 1.2 项目简介

**DBay 数据港湾** — 一个 Serverless 数据平台原型，包含三大模块：
- **Serverless PostgreSQL**：基于 Neon 存算分离架构，支持多版本、分支、时间旅行
- **知识库**：文档解析 + 向量检索 + 全文搜索 + 表知识库
- **数据湖**：Python/Ray 任务调度，支持数据导出和模型微调

**记忆库计划**：Neuromem 记忆库当前在独立项目中开发，计划未来合入 DBay，形成完整的 Agent 数据底座（知识库 + 记忆库 + 数据湖）。

## 2. 六个关键实践


### 2.1 实践一：人定目标，AI 结构化路线图

**核心理念**：你有产品愿景，AI 帮你拆解成可执行的阶段化计划。

#### 为什么重要

AI 没有产品判断力，但能帮你把模糊想法变成结构化的实施路线。关键是：
- 人负责定义"做什么"和"为什么"
- AI 负责拆解"怎么做"和"分几步"

#### 怎么做

1. **提出愿景**：我想做一个 Serverless PostgreSQL + 知识库 + 数据湖的平台
2. **和 AI 讨论技术可行性**：基于 Neon 存算分离、K8s 调度、向量检索
3. **输出阶段化路线图**：从 Stage 0 本地环境到 Stage 16 数据湖
4. **记入 Memory**：让每个新会话都知道项目全貌和当前进度

#### 真实案例

ROADMAP.md 的演进过程：
- **初版 (03-04)**：Stage 0-5（本地 → OBS → RDS → CCE → 控制台 → SRE）
- **第一次扩展 (03-06)**：加入 Stage 6b（PG 导入）
- **第二次扩展 (03-08)**：加入 Stage 10a-10e（备份、分支、审计、权限）
- **第三次扩展 (03-17)**：加入 Stage 11-12（时间旅行、弹性节点池）
- **最终版 (03-22)**：16 个阶段，覆盖全功能

**关键**：路线图不是一次定好的，是随着项目推进、技术验证、需求变化不断调整的。

### 2.2 实践二：Design Spec → Plan → Code 三段式

**核心理念**：AI 写代码很快但容易偏离方向，前期 review 文档比后期修代码成本低 10 倍。

#### 三段式流程

```
1. Design Spec (设计文档)
   ↓ 人 review 确认
2. Implementation Plan (实现计划)
   ↓ 人 review 确认
3. Code (编码实现)
   ↓ E2E 测试验收
```

#### 正面案例：Stage 15b 知识库增强

**时间线**：2026-03-19，1 天完成 4 个特性

1. **设计文档** (09:00-09:30)
   - `knowledge-enhancements-design.md` (778 行)
   - 定义 4 个特性：文档标签、查询重写、重排序、表知识库
   - 明确 API 接口、数据模型、技术选型

2. **实现计划** (09:30-10:00)
   - `knowledge-enhancements-plan.md` (778 行)
   - 拆解 8 个任务，标注依赖关系
   - 每个任务有明确的验收标准

3. **编码实现** (10:00-17:00)
   - 后端：JSONB 字段 + GIN 索引 + LLM 查询重写 + Reranker 集成
   - 前端：标签 UI + 聊天式搜索 + 表 KB 管理
   - 测试：48 个 E2E 测试全部通过

**结果**：一次性通过，无返工。

#### 反面案例：Stage 15 KB Pipeline

**时间线**：2026-03-18，反复调试浪费大量时间

**问题**：没有仔细审查设计文档就开始编码

1. AI 给的方案：用独立的 `kb-write` Pod 写入用户数据库
2. 实际问题：`kb-write` Pod 和用户的 `compute` Pod 同时写入，触发 Neon WAL 冲突
3. 调试过程：
   - 发现写入失败 → 查日志 → 发现 WAL 冲突
   - 重新设计方案 → 改为复用用户 compute pod 写入
   - 实现 `KbWriteQueue` 串行化写入
   - 再次测试 → 通过

**教训**：如果在设计阶段就和 AI 讨论"多个 Pod 同时写入 Neon 会怎样"，这个问题 30 分钟就能发现并修正方案。

#### 数据支撑

- Fix 提交占 27% (124/463)
- 很多 fix 是因为设计不清晰导致的返工
- 如果设计 review 更严格，这个比例可以降到 15% 以下

### 2.3 实践三：多窗口并行 + worktree 隔离

**核心理念**：AI 可以同时处理多个任务，人的瓶颈是注意力切换，不是 AI 的处理能力。

#### 日常工作方式

我通常开 **3-4 个 Claude Code 窗口**并行工作：

| 窗口 | 用途 | 示例 |
|------|------|------|
| A | 当前项目主特性开发 | Lakeon Stage 16 数据湖 |
| B | 同项目的另一个特性 (worktree) | Lakeon Stage 15c 切片管理 |
| C | 另一个项目 | 记忆库项目 MemOS 集成 |
| D | 调试/部署/运维 | CCE 环境问题排查 |

#### git worktree 用法

```bash
# 创建 worktree 用于特性开发
git worktree add .worktrees/feature-chunk-mgmt -b feature/chunk-mgmt

# 在 worktree 中开发（AI 在独立分支工作）
cd .worktrees/feature-chunk-mgmt
# ... AI 在这里编码 ...

# 完成后合并回主分支
git checkout main
git merge feature/chunk-mgmt
git worktree remove .worktrees/feature-chunk-mgmt
```

#### 避免打架的关键

1. **不同窗口不要改同一个文件** — 会产生冲突
2. **worktree 做特性开发，主目录做集成和部署** — 职责分离
3. **每个窗口有独立的 Memory context** — 避免上下文污染
4. **跨项目并行时，确保项目间无依赖** — 否则会阻塞

#### 实际效果

- 理论上 4 个窗口 = 4x 效率
- 实际效果 ≈ **2-3x**（有等待和协调成本）
- 但比单窗口串行快得多

#### 典型场景

**场景 1：同项目双特性并行**
- 窗口 A：开发 Dataset 导出功能（主分支）
- 窗口 B：开发知识库重排序（worktree）
- 两个特性互不依赖，完成后分别合并

**场景 2：跨项目并行**
- 窗口 A：Lakeon 数据湖开发
- 窗口 C：MemOS 记忆库集成
- 利用 AI 处理时间做另一个项目

**场景 3：开发 + 调试并行**
- 窗口 A：写新功能
- 窗口 D：排查 CCE 网络问题
- 调试等待时间不浪费

### 2.4 实践四：CLI 驱动的 E2E 测试闭环

**核心理念**：让 AI 能自己验证自己的工作，形成自动化调试循环。

#### 为什么 CLI 是关键

传统方式：
- AI 写代码 → 人手动测试 → 发现问题 → 告诉 AI → AI 修复 → 人再测试

CLI 驱动方式：
- AI 写代码 → AI 自己用 CLI 写测试 → AI 自己跑测试 → 失败 → AI 读错误 → AI 自己修 → 再跑 → 通过

**关键差异**：人从测试执行者变成验收者，AI 完成了整个调试循环。

#### 实施步骤

1. **开发 CLI 工具**：`dbay` CLI（本身也是产品功能）
   ```bash
   dbay db create mydb
   dbay db list
   dbay branch create mydb dev
   dbay version create mydb v1.0
   dbay kb upload mydb-kb document.pdf
   ```

2. **AI 用 CLI 写测试**：
   ```python
   def test_database_lifecycle():
       # AI 自己写的测试
       result = run_cli("dbay db create test-db")
       assert "created" in result
       
       result = run_cli("dbay db list")
       assert "test-db" in result
       
       # 连接测试
       conn = psycopg2.connect(get_connstr("test-db"))
       # ...
   ```

3. **自动调试循环**：
   ```
   AI 跑测试 → pytest 输出错误
   ↓
   AI 读错误信息："ConnectionRefusedError: compute pod not ready"
   ↓
   AI 分析：需要等待 compute pod 启动
   ↓
   AI 修改代码：添加 wait_for_compute_ready()
   ↓
   AI 再跑测试 → 通过 ✓
   ```

#### 真实案例：E2E 测试自动调试

**场景**：测试数据库创建后立即连接失败

```
FAILED tests/e2e/test_database.py::test_create_and_connect
E   psycopg2.OperationalError: could not connect to server
```

AI 的自动修复过程：
1. 读取错误 → 判断是 compute pod 未就绪
2. 修改 `DatabaseService.createDatabase()` → 添加等待逻辑
3. 重新跑测试 → 通过

**耗时**：约 3 分钟（如果是人工测试，至少 15 分钟）

#### 测试覆盖

48 个 E2E 测试覆盖：
- 数据库 CRUD (8 个)
- 分支管理 (15 个)
- 版本管理 (12 个)
- 知识库 (48 个)
- 数据湖 (23 个)
- 多租户隔离 (5 个)
- 认证授权 (3 个)

每次代码改动，AI 自动跑全部测试，快速回归。

### 2.5 实践五：Skill 沉淀 — 把经验变成 AI 的能力

**核心理念**：把踩坑经验固化为 AI 的行为规范，团队可共享复用。

#### 什么是 Skill

Skill 是 Claude Code 的扩展机制，可以把特定领域的知识、规范、工具用法封装成 AI 可调用的技能。

#### 我创建的 5 个 Skill

| Skill | 用途 | 价值 |
|-------|------|------|
| **huawei-cloud-style** | 华为云控制台 UI 风格 | 参考真实截图提取设计规范，AI 生成一致风格的前端 |
| **sre-console** | SRE 运维控制台模板 | 从实际运维中提炼标准页面结构（指标/日志/告警/资源） |
| **hcloud** | 华为云 CLI 操作 | 封装 hcloud CLI 用法，避免 AI 自己用 REST API 出错 |
| **deploy** | 部署前预检 | 防止凭据丢失、镜像 tag 错误等历史事故重演 |
| **weekly-report** | 周报自动生成 | 根据工作内容生成格式化周报 |

#### 案例 1：huawei-cloud-style Skill

**背景**：我希望 Console 和 Admin 的 UI 风格和华为云官方控制台一致。

**做法**：
1. 截图华为云 OBS、CCE、RDS 等服务的控制台
2. 让 AI 分析截图，提取设计规范：
   - 配色：深色侧边栏 #191919，主区域 #f5f5f5
   - 表格：斑马纹，hover 高亮，固定表头
   - 卡片：白色背景，1px 边框，8px 圆角
   - 按钮：主按钮蓝色，次按钮灰色
3. 写成 Skill 文档（357 行）
4. 之后每次让 AI 做前端，触发这个 Skill，自动应用华为云风格

**效果**：Console 和 Admin 的视觉风格高度一致，像是同一个设计师做的。

#### 案例 2：deploy Skill

**背景**：手动部署出过多次事故：
- `helm upgrade` 时 `$RDS_PRIVATE_IP` 为空 → RDS 配置丢失
- `kubectl set image` 创建字段管理器 → Helm 冲突
- 忘记跑测试就部署 → 线上 bug

**做法**：
1. 把所有部署事故总结成检查清单
2. 写成 deploy Skill（包含预检脚本）
3. AI 在部署前自动调用这个 Skill：
   - 检查环境变量是否为空
   - 强制跑测试
   - 确认镜像 tag 正确
   - 禁止手动 kubectl 操作

**效果**：部署事故从频繁发生降到 0。

#### Skill 的价值

1. **个人**：把踩坑经验固化，不会重复犯错
2. **团队**：Skill 可以共享，新人也能遵循最佳实践
3. **AI**：Skill 让 AI 具备领域知识，不需要每次都解释

### 2.6 实践六：用 /insight 让 AI 帮你改进工作方式

**核心理念**：AI 不只是写代码的工具，也是改进工作流程的顾问。

#### /insight 命令

Claude Code 的 `/insight` 命令可以让 AI 分析你的使用模式，提出改进建议。

#### 我采纳的改进

1. **创建 Skill 自动化重复性工作**

   - 建议：创建 deploy Skill 自动化预检

   - 结果：AI 输出质量明显提升

3. **优化 worktree 使用**
   - AI 建议用 worktree 隔离并行特性开发
   - 结果：多窗口并行效率从 1.5x 提升到 2-3x

#### 元认知

这是一种"元认知"能力 — AI 帮你反思自己的工作方式，而不只是执行任务。

## 3. 踩坑大全

### 3.1 AI 不了解的事（必须人判断）

| 类别 | 案例 | 教训 |
|------|------|------|
| **云服务兼容性** | pg_search BM25 与 Neon SMGR 不兼容 → 数据库 PANIC crash | AI 不知道底层存储引擎的实现细节 |
| **网络拓扑** | CCE overlay 网络不通 RDS（需要 hostNetwork）<br>CCI 不通 RDS/OBS（需要 VPC 对等连接） | AI 看不到你的 VPC 配置和路由表 |
| **跨境限制** | ISP 拦截 Railway（新加坡）→ 中国 IP 的 POST 请求 | AI 不知道中国网络政策和 GFW |
| **容器运行时限制** | CCE Autopilot setrlimit(CORE, INFINITY) 失败 | AI 不知道容器运行时的安全策略 |
| **云服务定价** | 误以为 CCI 按 Pod 数计费，实际按 CPU/内存时长 | AI 的定价信息可能过时 |

**结论**：涉及云环境、网络、安全策略的问题，必须人来判断和验证。

### 3.2 AI 常犯的错误（需要人 review）

| 错误类型 | 频率 | 典型案例 | 如何预防 |
|----------|------|----------|----------|
| **API 大小写不一致** | 高 | 后端返回 `RUNNING`，前端期望 `running` | 在设计文档里明确约定 |
| **环境变量为空不报错** | 严重 | `helm upgrade` 用 `$RDS_PRIVATE_IP`，变量为空导致配置丢失 | 部署脚本校验，禁止手动操作 |
| **测试 mock ≠ 真实行为** | 中 | 单元测试 mock 通过，E2E 测试失败 | E2E 测试兜底 |
| **忘记废弃旧字段** | 中 | 数据库加新字段，忘记删旧字段，导致数据不一致 | 设计文档标注 breaking changes |
| **TypeScript 类型错误** | 中 | 前端构建失败（strict mode 报错） | 开启 strict mode，AI 自己修 |

---

### 3.3 人常犯的错误（和 AI 协作时）

| 错误 | 后果 | 正确做法 |
|------|------|----------|
| **没 review 设计就让 AI 编码** | KB Pipeline 返工，浪费大量时间 | 花 30 分钟 review 设计文档 |
| **手动 helm upgrade** | RDS 配置丢失，API CrashLoopBackOff | 永远用 `deploy.sh` 脚本部署 |
| **长会话里混太多任务** | Context 污染，AI 输出变差 | 一个会话一个主题 |
| **信任 AI 的部署命令不检查** | 环境变量被覆盖，凭据泄露 | 部署前检查 diff |
| **跳过 E2E 测试直接部署** | 线上 bug，用户受影响 | E2E 测试强制门禁 |

### 4.3 AI 开发规范还需要补齐什么

**当前有的**：

- 项目级 Skill（部署、前端风格、SRE 模板等）
- Memory 系统（跨会话上下文）

**还缺少的**：
1. **AI 代码 review 标准** — 谁来 review AI 写的代码？用什么标准？
2. **安全审计流程** — AI 是否引入了安全漏洞？如何检测？
3. **可维护性评估** — AI 写的代码别人能看懂吗？注释够吗？
4. **性能基准** — AI 写的代码性能如何？有没有明显的性能问题？
5. **容灾测试** — 故障转移、数据恢复是否可靠？
6. **CI/CD 集成** — 如何把 AI 开发流程集成到现有 CI/CD？

**AI 大幅加速了原型开发**，但产品化需要：
- 团队协作（安全、运维、产品、运营）
- 合规和流程规范适应 AI 开发模式
- 建立 AI 代码的质量保障体系
