Why echomem

三家 Agent 各自记笔记，
切一次工具就丢一次上下文。

日常在 Claude Code、openclaw、hermes 之间切换。同一个事实、同一份 skill、同一段会话——换个 Agent 就要重讲一遍。echomem 让它们共享同一份本地大脑。

痛点

四个具体问题

下面四个场景在当下 AI Agent 日常中高频出现，但没有一个好的本地方案能同时解决。

01

跨 Agent 记忆断裂

Claude Code 里讨论过的架构决策，切到 openclaw 写代码时丢了。同一件事要在三家 Agent 分别重述，浪费时间也丢失精度。

02

离线场景受阻

同事 demo、出差路上、客户现场——没有外网时，所有"云记忆"方案直接失效。本地不能躺在一台 MacBook 上给 Agent 提供完整的记忆服务。

03

衍生物依赖云端 pipeline

时间线、知识图谱、摘要——这些"让你的记忆活起来"的能力，现有方案绑死云 GPU。本地小模型（gemma 4B）找不到对应实现。

04

研究工作侵入主线

"用记忆预测 LLM 输出长度"等研究假设需要实验通道，但不能动现有 Agent 业务代码。好的探索需要好隔离。

想验证"记忆衍生物对 Agent 产出质量的影响"这类假设？echomem 给你一个完全本地的实验环境——不动你的 Agent 代码，hook 接入即用。

目标

三个量化目标

以下目标依赖 本地 memory + 本地小模型推理——不假设云端 GPU、不依赖外部 API。

01

跨 Agent 记忆共享

OpenClaw · Hermes Agent · OfficeClaw · ClaudeCode 等多个 Agent 通过统一 API 读写同一份记忆池，切换工具不丢上下文。

02

长程记忆评测领先

LoCoMo 任务得分达到 90%，LongMemEval 和 EvolvingEvents 基准处于业界前列。用本地模型追上云端方案的表现。

03

TTFT 降低 2 倍

OpenClaw 多轮交互的首 Token 延迟（TTFT）降低 2 倍。记忆缓存命中时跳过远端 embedding 调用，本地直接返回上下文。

"不是又一个 memory benchmark 刷分工具。而是在真实多 Agent 工作流里，用本地小模型做到云端级别的记忆体验。"

设计原则

六条不动摇的取舍

Local-first

本地是主战场。备份就是 cp ~/.echomem。云端 dbay 是可选挂载，不是默认依赖。

同源不重写

schema 与 API 概念尽量与云端 dbay 兼容。StorageDriver 抽象同时支持 SQLite / PostgreSQL / Cloud。

存储抽象层

所有上层只见 StorageDriver 接口。切换底层不动业务代码——这是同源策略的兜底。

小模型友好

衍生物 pipeline 对 gemma 4B 量级模型留降级路径。置信度低不污染正式数据，进 pending 表。

零云依赖默认

demo 必须在没外网时跑通。云挂载是用户主动行为，需要显式开启。

Hook 不入侵

Insight Track 等可选能力通过 Agent 框架 hook 接入，不修改 Agent 业务代码。

"echomem 不是一个新东西。它是把 dbay 已经验证过的那些概念——记忆、衍生物、AgentFS——挪回本地，让一台 MacBook 也能体面地服务三家 Agent。"
