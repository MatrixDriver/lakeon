架构方案（基于视频 pipeline 用例修正）

核心挑战：fan-out（1→N 裂变）、条件分支、可暂停节点

验收用例的执行模型

原始视频 (10min, 1个文件) │ ▼ 【规整适配】 — 纯函数，内存传递 │ ▼ 【视频切片】 — fan-out: 1→42 clips │ ▼ ×42 并行 【规则清洗】 │ ├─ 通过 → 标记 PASS (4~5个) │ ├─ 淘汰 → 标记 DROP │ └─ 需裁剪 → 条件分支 │ ▼ │ 【裁剪处理】 │ ├─ 裁剪后通过 → 标记 PASS │ └─ 裁剪后淘汰 → 标记 DROP │ ▼ fan-in: 合并所有 PASS clips │ ▼ 并行 【模型清洗】 (VQA/水印/字幕/光流) │ ▼ 【质检】 — 可配置：模型自动 或 人工暂停 │ ▼ 【内容标注】 (VICLIP/Caption/运镜) │ ▼ 发布 (3 clips, 留存 12.1%)
A

Ray DAG + 自研调度层

lakeon-api 中实现轻量 DAG 调度器，负责 fan-out/条件分支/暂停恢复的编排逻辑；每个步骤的实际计算提交到 Ray 集群执行

调度器 lakeon-api（Java）管理 DAG 状态机，步骤间路由、fan-out/in、暂停
计算执行 每个步骤是 Ray remote function，同一 Ray 集群内执行
数据传递 步骤间 Ray object store（内存），checkpoint 节点写 OBS
暂停恢复 调度器标记步骤为 WAITING_APPROVAL，Ray Job 结束释放资源；人工确认后启动新 Ray Job 从 checkpoint 恢复

优势

  • 调度逻辑在 Java 侧，完全可控，fan-out/条件分支自由实现
  • 暂停时释放 Ray 资源，不浪费算力
  • DAG 状态持久化在 RDS，天然可靠
  • 实时 DAG 视图直接读 RDS 状态，无需适配 Ray Dashboard
  • 复用现有 RayJobRunner

挑战

  • 需要自研 DAG 状态机（中等工作量）
  • 暂停恢复时需要从 OBS checkpoint 重建上下文

推荐 — 调度归 Java（可靠+可控），计算归 Ray（高性能），职责清晰

B

纯 Ray Workflow

整条 pipeline 翻译为一个 Ray Workflow，所有编排逻辑（fan-out/条件/暂停）都在 Ray 内部处理

调度器 Ray Workflows（内置 DAG 编排）
计算执行 Ray remote functions
暂停恢复 Ray Workflow 原生支持，但 Ray 集群需保持运行

优势

  • 内置断点续跑和重试
  • 全程内存传递，无需手动管理 checkpoint

挑战

  • Ray Workflows API 仍为 alpha，生产风险高
  • 暂停等待人工审核时 Ray 集群不能释放，浪费资源
  • fan-out 后的条件分支需要大量 Python 胶水代码
  • DAG 状态在 Ray 内部，前端实时视图需要额外适配
  • YAML → Ray Workflow 翻译复杂度高
C

每步独立 Ray Job

每个 pipeline 步骤作为独立 Ray Job 提交，lakeon-api 管理步骤间的串联和数据流转

调度器 lakeon-api（Java）逐步提交
计算执行 每步独立 Ray Job（独立集群生命周期）
数据传递 步骤间必须走 OBS，无法内存传递

优势

  • 最简单，完全复用现有 Job 基础设施
  • 步骤间天然隔离，暂停恢复简单

挑战

  • 每步启停 Ray 集群，开销大（与之前选 C 函数级的决策矛盾）
  • 所有数据走 OBS，视频数据 IO 成为瓶颈
  • 42 个 clips 并行 = 42 个 Ray Job？资源管理复杂
