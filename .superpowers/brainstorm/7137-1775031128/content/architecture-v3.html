数据生产线架构方案对比

核心挑战：fan-out（1→N）、条件分支、可暂停节点

视频 Pipeline 验收用例的执行流

📹 原始视频 (10min, 1个文件)
↓ 【规整适配】纯函数，内存传递
↓ 【视频切片】fan-out: 1 → 42 clips
↓ ×42 并行【规则清洗】
✓ 通过 → PASS (4~5个)
✗ 淘汰 → DROP
⚡ 需裁剪 → 【裁剪】→ PASS 或 DROP
↓ fan-in: 合并所有 PASS clips
↓ 并行【模型清洗】VQA / 水印 / 字幕 / 光流
↓ 【质检】可配置：模型自动 or 人工暂停
↓ 【内容标注】VICLIP / Caption / 运镜
↓ 发布 (3 clips, 留存 12.1%)
A

Ray DAG + 自研调度层 ✦ 推荐

Java 管编排，Ray 管计算。
lakeon-api 中实现轻量 DAG 调度器，负责步骤路由、fan-out/in、条件分支、暂停恢复。
每组连续的计算步骤打包为一个 Ray Job 提交执行。

调度器 lakeon-api (Java) — DAG 状态机，持久化在 RDS
计算 Ray remote functions，同一集群内存传递
数据流 步骤间 Ray object store；checkpoint 写 OBS
暂停 标记 WAITING → 释放 Ray 资源 → 确认后新 Ray Job 从 OBS 恢复
优势
  • fan-out/条件分支在 Java 中自由实现
  • 暂停时释放 Ray，不浪费算力
  • DAG 状态在 RDS，天然可靠
  • 前端实时视图直接读 RDS
  • 复用现有 RayJobRunner
挑战
  • 需自研 DAG 状态机（中等工作量）
  • 暂停恢复需从 OBS checkpoint 重建
B

纯 Ray Workflow

整条 pipeline 翻译为一个 Ray Workflow，
所有编排（fan-out/条件/暂停）都在 Ray 内部处理。

优势
  • 内置断点续跑和重试
  • 全程内存传递
风险
  • Ray Workflows API 仍为 alpha
  • 暂停时集群不能释放，浪费算力
  • YAML → Ray Workflow 翻译复杂
  • 前端状态需额外适配 Ray Dashboard
C

每步独立 Ray Job

每个步骤独立提交为 Ray Job，
lakeon-api 逐步串联，步骤间数据走 OBS。

优势
  • 最简单，复用现有 Job 基础设施
  • 步骤间天然隔离
风险
  • 每步启停集群，开销大
  • 所有数据走 OBS，视频 IO 瓶颈
  • 42 clips 并行 = 42 个 Job？
  • 与"函数级组件"决策矛盾
