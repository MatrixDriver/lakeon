数据生产线架构方案对比

选择 pipeline 执行引擎的核心架构

A

Ray DAG Native — 纯 Ray 方案

Pipeline 定义（YAML）→ lakeon-api 翻译为 Ray DAG → 提交为单个 Ray Job → Ray 内部调度所有步骤

优势

  • 零拷贝内存传递，性能最优
  • Ray 原生 DAG 调度，成熟稳定
  • 完全复用现有 RayJobRunner 基础设施
  • 组件就是 Python 函数，开发门槛低
  • checkpoint 用 Ray object store + OBS 混合

挑战

  • 步骤粒度的资源分配不如独立 Pod 灵活
  • 单个 Ray 集群故障影响整条 pipeline
  • 需要自研 YAML → Ray DAG 的翻译层

推荐 — 最契合"函数级组件 + Ray 执行"的决策，架构最简洁

B

Hybrid — Ray + 独立 Job 混合

轻量步骤在 Ray DAG 内执行，重型步骤（如大模型推理、微调）单独启动 GPU Job Pod，pipeline 引擎在 lakeon-api 中编排

优势

  • 重型步骤可独立申请 GPU 资源
  • 局部失败不影响整条 pipeline
  • 可以混用 Python Job + Ray Job

挑战

  • 两种执行模型，编排逻辑复杂
  • 跨 Job 数据传递必须走 OBS，性能下降
  • lakeon-api 需要自研 DAG 调度器
  • 与"函数级组件在同一 Ray Job 内执行"的决策不一致
C

Ray Workflows — 基于 Ray Workflows API

使用 Ray 内置的 Workflows 库（持久化 DAG），天然支持断点续跑、步骤级重试、容错恢复

优势

  • 内置持久化，步骤失败可从断点恢复
  • 内置重试策略和超时
  • 步骤级状态追踪（天然支持实时 DAG 视图）
  • 仍然是单 Ray 集群，内存传递高效

挑战

  • Ray Workflows 仍标记为 alpha/beta，API 可能变化
  • 持久化依赖 Ray 内部存储，需确保与 OBS 对齐
  • 调试和监控需要额外适配 Ray Dashboard
  • YAML → Ray Workflow 翻译比方案 A 更复杂
