# Lakeon Design Context

本文件由 `/impeccable teach` 生成，作为 Lakeon 前端设计的约束源。后续所有 impeccable 系列 skill（craft、layout、typeset、critique、polish 等）在进入工作前必须先读取本文件。

---

## Users

**谁在用**：

- **Console**（`lakeon-console`）：自托管 Serverless PostgreSQL 的付费/试用开发者 —— 数据库工程师、AI 应用开发者、数据平台负责人。会在 IDE 和浏览器之间来回切换，一天可能进来 3–20 次，每次处理具体事务：建库、跑查询、上传知识库文档、看记忆库、调 Notebook。
- **Admin**（`lakeon-admin`）：Lakeon 团队自己的 SRE 运维人员。处理集群健康、Pod 重启、日志追踪、租户管理等后台操作。
- **共同特征**：都是有经验的技术人员，不需要被"教育"，反感冗余文案、不需要 emoji 和吉祥物，厌恶浮夸营销话术。

**在什么场景下用**：

- 日间自然光办公室、IDE 旁边一个浏览器 tab
- 偶尔在夜晚值守 / oncall（Admin 尤其明显）
- 屏幕尺寸以 13"–27" 为主，不优先考虑手机

**要完成的工作（Jobs to be Done）**：

1. "我需要一个能秒开的 Serverless PG，别让我去管 K8s"
2. "我想把一堆文档变成可检索的知识库，看到从上传→解析→向量化→检索的全过程"
3. "我想看到我的记忆库里存了什么、被召回了什么、为什么"
4. "出问题了，我要能立刻看到日志、重启 Pod、定位租户"

## Brand Personality

**三个词**：**技术 · 温润 · 匠气**

拆解：

- **技术**（Technical）：专业主义是第一性的。界面要让用户相信"这些人真的懂他们在做什么"。数据是主角，文案为数据服务。不卖弄 AI 概念、不堆砌形容词。
- **温润**（Warm & calm）：不是冷静的银灰金属感，而是像木质家具、港湾黄昏、手抄笔记本那种温度。色彩有暖意，留白有呼吸，细节不喧宾夺主。
- **匠气**（Crafted）：每一个间距、字重、对齐都是被想过的。能看出是"有人做的"，不是"生成的"。允许有一点刻意、一点编辑感、一点手作感。

**情绪目标**：信任、从容、被尊重的专业感。用户应该感到"这个工具可靠、舒服、我愿意长期用"，而不是"这个工具看起来很酷"。

## Aesthetic Direction

### 核心方向

**港湾编辑室（Harbor Editorial）** —— 想象一间落地窗开向港湾的独立工作室：

- 地板是木色，墙壁是暖白，海面反射的蓝是深邃但不冷的藏青
- 手边摆着一本 1970 年代的技术手册和一台老式打字机
- 桌面是清空的：只有当前在处理的那一份文件
- 一杯凉掉的咖啡留下一个暖橙色的圆环作为唯一的装饰

这个意象解释了所有后续的设计决策：为什么色彩是暖的而不是冷的，为什么标题用衬线而正文不用，为什么留白比装饰重要，为什么不需要任何插图或吉祥物。

### 主题：Light only（至少 v1）

用户是白天办公室场景下的技术人，不是夜晚交易员也不是影音消费者。暖色调默认用 light 主题。以后若 Admin 的夜间值守场景明确成需求，再单独给 Admin 做一套深色，但 **不是** 霓虹荧光发光那种——而是把当前暖色调倒置到深藏青 + 暖米文字。

### 色彩

当前已有一套"港湾暖色调"变量（见 `lakeon-console/src/style.css`），已经合并 main 并被用户认可，继承而非推倒：

| 角色 | 现值 | 含义 |
|---|---|---|
| `--c-primary` | `#2a4d6a` | 深港湾蓝（藏青） |
| `--c-primary-hover` | `#1e3a52` | 悬停加深 |
| `--c-accent` | `#c67d3a` | 港湾暖橙（咖啡印子） |
| `--c-accent-text` | `#9a5b25` | 暖橙文字色，保证对比度 |
| `--c-accent-light` | `#fdf5ed` | 暖橙 10% 背景 |
| `--c-bg-alt` | `#faf8f5` | 暖米背景（木色地板） |
| `--c-text` | `#2c3e50` | 主文字，偏蓝灰不纯黑 |
| `--c-border` | `#e8e4df` | 暖灰边框 |

**演进方向**（优先级 P2，非本次 teach 的范围）：

- 将 hex 值逐步迁移到 **OKLCH**，让明度阶梯感知均匀
- 将中性色（灰/边框/背景）显式向品牌橙 hue 微调 `chroma ≈ 0.005–0.008`，强化潜意识呼应
- 深化"暖橙作为稀有强调色"的 60-30-10 纪律：暖橙只用在主 CTA、焦点态、关键链接，不在卡片边框/图标/大块背景里滥用

**绝对禁区**：

- 紫→蓝 gradient、青色 on 黑、荧光霓虹（用户明确反对 Neon/Supabase 美学）
- 政务 SaaS 风格的灰蓝密集表格 + 厚重按钮（用户明确反对华为云/阿里云政务风。注意：本项目过去的 `sre-console`、`huawei-cloud-style` skill 是**遗留**的，在新设计中优先级低于本文件）
- 纯黑 `#000` / 纯白 `#fff`
- 左侧色条边框（`border-left: 3px+`）作装饰——这是 impeccable 的绝对禁令
- 任何 gradient text（`background-clip: text` + 渐变）

### 字体

**当前使用 DM Sans，必须替换** —— DM Sans 是 impeccable reflex font 禁用列表之一，也是全球 AI 生成 UI 最易辨识的指纹之一。

**推荐方案**（显示衬线 + 中性无衬线正文，与用户选择一致）：

- **显示/标题**：**Source Serif 4**（Adobe，Google Fonts 可得）
  - 为什么：1990 年 Adobe Source 家族的传承，Frank Grießhammer 现代化重制，有编辑室的暖意和文献的匠气；可变字重从 Light 到 Black；**不在** reflex 禁用表里；与 Fraunces/Newsreader/Playfair/Cormorant 的"文艺咖啡馆默认"明确区分。
  - 候补：**Literata**（Google Fonts 可得，可变字重，学术书籍感更强）
- **正文/UI**：**Geist Sans**（Vercel 出品，开源，Google Fonts + npm `geist` 可得）
  - 为什么：几何精确但字形温润，metrics 非常成熟适合密集 UI，不在 reflex 禁用表里；技术感与温润感平衡得恰到好处。
  - 候补：**Manrope**（Google Fonts 可得，圆润几何）
- **等宽**：**JetBrains Mono** 或 **Geist Mono**（用于 SQL、代码块、API key 展示。注意 IBM Plex Mono 和 Space Mono 已被禁用）

**使用规则**：

- 页面标题、卡片标题、Hero 文案 → Source Serif 4（weight 400/500/600）
- 正文、表格、按钮、Label → Geist Sans（weight 400/500/600）
- 代码、ID、Token、终端输出 → JetBrains Mono
- **禁止**：在大段正文（>1 行）使用等宽、在长文案使用全大写、在任何地方使用 gradient text

### 空间与节奏

- 4pt 间距阶梯，语义命名（`--space-xs/sm/md/lg/xl/2xl`），不用像素名
- 使用 `gap` 而非 margin 处理兄弟间距
- 刻意制造节奏：标题上方空间 ≥ 下方空间 1.5×；关键信息周围留白比装饰更有力
- Cards 不嵌套 Cards；不是所有内容都需要卡片包裹
- 表格是本产品的主角之一，表格的留白、行高、斑马条（不用！）、hover 态必须特别打磨

### 运动

- 使用 ease-out-quart/quint 的指数缓动
- 只动 `transform` 和 `opacity`
- 一个精心编排的"页面进入"微动效胜过 10 个散落的 micro-interaction
- 禁止 bounce/elastic 缓动

## Accessibility

- 目标 **WCAG AA**（AAA 在关键 CTA 上努力达到）
- 所有色彩组合需通过 4.5:1 对比度（小字）/ 3:1（大字与图标）
- 尊重 `prefers-reduced-motion`，在该媒体查询下关闭所有非必要过渡
- 焦点态必须可见且明确：用 `--c-accent` 暖橙描边 + 2px offset
- 键盘导航：所有表格操作、Dialog、表单都必须支持
- 色彩不能是唯一的信息载体（例：状态既有色点也有文字）

## Dual-App Consistency

**Console 与 Admin 共享同一设计语言**，差异仅在：

- **数据密度**：Admin 的表格行高更紧（行高 36px vs Console 48px），字号更小（13px vs 14px），一屏展示更多行
- **导航深度**：Admin 侧边栏允许更深层级（2–3 层），Console 保持扁平
- **颜色侧重**：Admin 允许更频繁使用 `--c-danger` 红色（错误、告警）和 `--c-primary` 深蓝（资源 ID、主操作）；Console 则让 `--c-accent` 暖橙更多承担主 CTA
- **不共享**：各自的布局 shell（header、sidebar、breadcrumb）不强行抽成同一组件，避免耦合

两边的 CSS 变量应该完全一致，按钮、表格、Dialog、Form 的视觉应该像素级对齐。

## Design Principles（五条可操作的纪律）

未来每一个界面决策都应该能被这五条验证：

1. **数据为主，文案为辅** —— 界面是用来展示和操作数据的，不是展示品牌声音的。任何不服务于当前数据/操作的文字、图标、装饰都应该被砍掉。
2. **温度来自色彩与留白，不来自装饰** —— 暖的感觉用暖米背景、暖橙焦点、手写感衬线标题来实现，而不是用吉祥物、emoji、插图、gradient。
3. **节奏胜过对齐** —— 宁可留不规则但有呼吸的留白，也不要死板平均的 card grid。每个页面应该有一个"主角区域"被刻意放大。
4. **可见的选择胜过安全的默认** —— 字体不用 Inter/DM Sans，色彩不用纯黑纯白，标题字重大胆，正文克制。让挑剔的用户能看出"这是被想过的"。
5. **绝不出现 AI 生成的指纹** —— 禁止：左侧色条、gradient text、紫蓝渐变、大尺寸圆角图标列、Hero 大数字 + 小标签模板、卡片套卡片、全页使用单一字体家族。

## Known Constraints

- **技术栈**：Vue 3 + TypeScript + Vite + Pinia；已有 TinyVue 组件库基础（可用但不必完全依赖其视觉默认）
- **部署**：Console/Admin 部署在 Railway，git push 自动触发
- **兼容**：需支持 Chromium/Safari/Firefox 最新两版；不需要支持 IE
- **字体加载**：Source Serif 4 和 Geist 通过 `@fontsource/*` 或 Google Fonts 自托管，避免 CDN 单点和隐私问题；加 `font-display: swap` 避免白屏
- **非商业化内容**：网站与控制台不出现价格、订阅、升级等商业化文案（用户明确要求）
- **不要 emoji**：任何地方都不要用 emoji 当图标
