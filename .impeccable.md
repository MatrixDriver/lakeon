# DBay Design Context

本文件由 `/impeccable teach` 生成，作为 DBay 前端设计的约束源。后续所有 impeccable 系列 skill（craft、layout、typeset、critique、polish 等）在进入工作前必须先读取本文件。

> **命名约定**：对外产品名是 **DBay**（公共页面、Landing、Product、Docs、界面文案里都用 DBay）。`lakeon-console / lakeon-admin / lakeon-api / lakeon` 只是代码仓库的内部代号，不出现在任何面向用户的位置。本文件里的 "DBay" 都指产品，不是代号。

---

## Users

**谁在用**：

- **Console**（`lakeon-console`）：自托管 Serverless PostgreSQL 的付费/试用开发者 —— 数据库工程师、AI 应用开发者、数据平台负责人。会在 IDE 和浏览器之间来回切换，一天可能进来 3–20 次，每次处理具体事务：建库、跑查询、上传知识库文档、看记忆库、调 Notebook。
- **Admin**（`lakeon-admin`）：DBay 团队自己的 SRE 运维人员。处理集群健康、Pod 重启、日志追踪、租户管理等后台操作。
- **共同特征**：都是有经验的技术人员，不需要被"教育"，反感冗余文案、不需要 emoji 和吉祥物，厌恶浮夸营销话术。

**在什么场景下用**：

- 日间自然光办公室、IDE 旁边一个浏览器 tab
- 偶尔在夜晚值守 / oncall（Admin 尤其明显）
- 屏幕尺寸以 13"–27" 为主，不优先考虑手机

**要完成的工作（Jobs to be Done）**：

1. "我需要一个能秒开的 Serverless PG，别让我去管 K8s"
2. "我想把一堆文档变成可检索的知识库，看到从上传→解析→向量化→检索的全过程"
3. "我想看到我的记忆库里存了什么、被召回了什么、为什么"
4. "出问题了，我要能立刻看到日志、重启 Pod、定位租户"

## Brand Personality

**三个词**：**技术 · 温润 · 匠气**

拆解：

- **技术**（Technical）：专业主义是第一性的。界面要让用户相信"这些人真的懂他们在做什么"。数据是主角，文案为数据服务。不卖弄 AI 概念、不堆砌形容词。
- **温润**（Warm & calm）：不是冷静的银灰金属感，而是像木质家具、港湾黄昏、手抄笔记本那种温度。色彩有暖意，留白有呼吸，细节不喧宾夺主。
- **匠气**（Crafted）：每一个间距、字重、对齐都是被想过的。能看出是"有人做的"，不是"生成的"。允许有一点刻意、一点编辑感、一点手作感。

**情绪目标**：信任、从容、被尊重的专业感。用户应该感到"这个工具可靠、舒服、我愿意长期用"，而不是"这个工具看起来很酷"。

## Aesthetic Direction

### 核心方向

**港湾编辑室（Harbor Editorial）** —— 想象一间落地窗开向港湾的独立工作室：

- 地板是木色，墙壁是暖白，海面反射的蓝是深邃但不冷的藏青
- 手边摆着一本 1970 年代的技术手册和一台老式打字机
- 桌面是清空的：只有当前在处理的那一份文件
- 一杯凉掉的咖啡留下一个暖橙色的圆环作为唯一的装饰

这个意象解释了所有后续的设计决策：为什么色彩是暖的而不是冷的，为什么标题用衬线而正文不用，为什么留白比装饰重要，为什么不需要任何插图或吉祥物。

### 主题：Light only（至少 v1）

用户是白天办公室场景下的技术人，不是夜晚交易员也不是影音消费者。暖色调默认用 light 主题。以后若 Admin 的夜间值守场景明确成需求，再单独给 Admin 做一套深色，但 **不是** 霓虹荧光发光那种——而是把当前暖色调倒置到深藏青 + 暖米文字。

### 色彩

当前已有一套"港湾暖色调"变量（见 `lakeon-console/src/style.css`），已经合并 main 并被用户认可，继承而非推倒：

| 角色 | 现值 | 含义 |
|---|---|---|
| `--c-primary` | `#2a4d6a` | 深港湾蓝（藏青） |
| `--c-primary-hover` | `#1e3a52` | 悬停加深 |
| `--c-accent` | `#c67d3a` | 港湾暖橙（咖啡印子） |
| `--c-accent-text` | `#9a5b25` | 暖橙文字色，保证对比度 |
| `--c-accent-light` | `#fdf5ed` | 暖橙 10% 背景 |
| `--c-bg-alt` | `#faf8f5` | 暖米背景（木色地板） |
| `--c-text` | `#2c3e50` | 主文字，偏蓝灰不纯黑 |
| `--c-border` | `#e8e4df` | 暖灰边框 |

**演进方向**（优先级 P2，非本次 teach 的范围）：

- 将 hex 值逐步迁移到 **OKLCH**，让明度阶梯感知均匀
- 将中性色（灰/边框/背景）显式向品牌橙 hue 微调 `chroma ≈ 0.005–0.008`，强化潜意识呼应
- 深化"暖橙作为稀有强调色"的 60-30-10 纪律：暖橙只用在主 CTA、焦点态、关键链接，不在卡片边框/图标/大块背景里滥用

**绝对禁区**：

- 紫→蓝 gradient、青色 on 黑、荧光霓虹（用户明确反对 Neon/Supabase 美学）
- 政务 SaaS 风格的灰蓝密集表格 + 厚重按钮（用户明确反对华为云/阿里云政务风。注意：本项目过去的 `sre-console`、`huawei-cloud-style` skill 是**遗留**的，在新设计中优先级低于本文件）
- 纯黑 `#000` / 纯白 `#fff`
- 左侧色条边框（`border-left: 3px+`）作装饰——这是 impeccable 的绝对禁令
- 任何 gradient text（`background-clip: text` + 渐变）

### 字体

**当前使用 DM Sans，必须替换** —— DM Sans 是 impeccable reflex font 禁用列表之一，也是全球 AI 生成 UI 最易辨识的指纹之一。

**推荐方案**（显示衬线 + 中性无衬线正文，与用户选择一致）：

- **显示/标题**：**Source Serif 4**（Adobe，Google Fonts 可得）
  - 为什么：1990 年 Adobe Source 家族的传承，Frank Grießhammer 现代化重制，有编辑室的暖意和文献的匠气；可变字重从 Light 到 Black；**不在** reflex 禁用表里；与 Fraunces/Newsreader/Playfair/Cormorant 的"文艺咖啡馆默认"明确区分。
  - 候补：**Literata**（Google Fonts 可得，可变字重，学术书籍感更强）
- **正文/UI**：**Geist Sans**（Vercel 出品，开源，Google Fonts + npm `geist` 可得）
  - 为什么：几何精确但字形温润，metrics 非常成熟适合密集 UI，不在 reflex 禁用表里；技术感与温润感平衡得恰到好处。
  - 候补：**Manrope**（Google Fonts 可得，圆润几何）
- **等宽**：**JetBrains Mono** 或 **Geist Mono**（用于 SQL、代码块、API key 展示。注意 IBM Plex Mono 和 Space Mono 已被禁用）

**使用规则**：

- 页面标题、卡片标题、Hero 文案 → Source Serif 4（weight 400/500/600）
- 正文、表格、按钮、Label → Geist Sans（weight 400/500/600）
- 代码、ID、Token、终端输出 → JetBrains Mono
- **禁止**：在大段正文（>1 行）使用等宽、在长文案使用全大写、在任何地方使用 gradient text

### 空间与节奏

- 4pt 间距阶梯，语义命名（`--space-xs/sm/md/lg/xl/2xl`），不用像素名
- 使用 `gap` 而非 margin 处理兄弟间距
- 刻意制造节奏：标题上方空间 ≥ 下方空间 1.5×；关键信息周围留白比装饰更有力
- Cards 不嵌套 Cards；不是所有内容都需要卡片包裹
- 表格是本产品的主角之一，表格的留白、行高、斑马条（不用！）、hover 态必须特别打磨

### 运动

- 使用 ease-out-quart/quint 的指数缓动
- 只动 `transform` 和 `opacity`
- 一个精心编排的"页面进入"微动效胜过 10 个散落的 micro-interaction
- 禁止 bounce/elastic 缓动

## Accessibility

- 目标 **WCAG AA**（AAA 在关键 CTA 上努力达到）
- 所有色彩组合需通过 4.5:1 对比度（小字）/ 3:1（大字与图标）
- 尊重 `prefers-reduced-motion`，在该媒体查询下关闭所有非必要过渡
- 焦点态必须可见且明确：用 `--c-accent` 暖橙描边 + 2px offset
- 键盘导航：所有表格操作、Dialog、表单都必须支持
- 色彩不能是唯一的信息载体（例：状态既有色点也有文字）

## Dual-App Consistency

**Console 与 Admin 共享同一设计语言**，差异仅在：

- **数据密度**：Admin 的表格行高更紧（行高 36px vs Console 48px），字号更小（13px vs 14px），一屏展示更多行
- **导航深度**：Admin 侧边栏允许更深层级（2–3 层），Console 保持扁平
- **颜色侧重**：Admin 允许更频繁使用 `--c-danger` 红色（错误、告警）和 `--c-primary` 深蓝（资源 ID、主操作）；Console 则让 `--c-accent` 暖橙更多承担主 CTA
- **不共享**：各自的布局 shell（header、sidebar、breadcrumb）不强行抽成同一组件，避免耦合

两边的 CSS 变量应该完全一致，按钮、表格、Dialog、Form 的视觉应该像素级对齐。

## Design Principles（五条可操作的纪律）

未来每一个界面决策都应该能被这五条验证：

1. **数据为主，文案为辅** —— 界面是用来展示和操作数据的，不是展示品牌声音的。任何不服务于当前数据/操作的文字、图标、装饰都应该被砍掉。
2. **温度来自色彩与留白，不来自装饰** —— 暖的感觉用暖米背景、暖橙焦点、手写感衬线标题来实现，而不是用吉祥物、emoji、插图、gradient。
3. **节奏胜过对齐** —— 宁可留不规则但有呼吸的留白，也不要死板平均的 card grid。每个页面应该有一个"主角区域"被刻意放大。
4. **可见的选择胜过安全的默认** —— 字体不用 Inter/DM Sans，色彩不用纯黑纯白，标题字重大胆，正文克制。让挑剔的用户能看出"这是被想过的"。
5. **绝不出现 AI 生成的指纹** —— 禁止：左侧色条、gradient text、紫蓝渐变、大尺寸圆角图标列、Hero 大数字 + 小标签模板、卡片套卡片、全页使用单一字体家族。

## Known Constraints

- **技术栈**：Vue 3 + TypeScript + Vite + Pinia；已有 TinyVue 组件库基础（可用但不必完全依赖其视觉默认）
- **部署**：Console/Admin 部署在 Railway，git push 自动触发
- **兼容**：需支持 Chromium/Safari/Firefox 最新两版；不需要支持 IE
- **字体加载**：Source Serif 4 和 Geist 通过 `@fontsource/*` 或 Google Fonts 自托管，避免 CDN 单点和隐私问题；加 `font-display: swap` 避免白屏
- **非商业化内容**：网站与控制台不出现价格、订阅、升级等商业化文案（用户明确要求）
- **不要 emoji**：任何地方都不要用 emoji 当图标

## Public Site · 品牌与 Landing 叙事（Phase A Brief）

以下约束专门用于 lakeon-console 的公开页（Landing / Product × 5 / 架构单页 / 导航）。不适用于 Blog、Docs（本轮不动），也不适用于 Console/Admin 的 app UI。

### 0. 核心真相

> **DBay 是 Agent 工作态和学习态的数据基础设施。**

- **工作态**（Working state）= Agent 执行任务时读写的数据：事务（Lakebase）、知识（Knowledge.search）、记忆（Memory.recall）
- **学习态**（Learning state）= Agent 执行完反刍的数据：批处理（Datalake）、反思整合（Memory.digest）、知识沉淀（Knowledge.curate）
- **数据基础设施** = substrate，不是 app / SDK / 平台。别的东西从这之上长出来。

所有 Landing 叙事围绕 **"一体 substrate vs 四件事拼装"** 展开。DBay **不是** 另一个托管 Postgres、更聪明的 Pinecone、更轻的 Ray —— 它是让 Agent 的**状态库 · 知识库 · 记忆库 · 数据湖同步加工**四件事发生在**同一套基础设施**上。对用户的直接感受是：弹性秒级扩缩、完全托管免运维，Agent 可以大胆探索试错，也能从任意断点恢复继续。（内部实现上建立在 Neon 的存算分离之上，Neon 是 DBay 的内核底座，不是竞品。技术实现细节 — copy-on-write / serverless / 多租户 — 留到 Screen 06 架构内核屏讲，不出现在 hero lede 上。）

### 1. 目标读者与次级入口

**Primary · CTO / 技术决策人 / 架构师** —— 正在给公司技术栈选型的人。

他们的心智：
- 不在乎花哨。要的是"靠不靠谱"、"5 年后还在不在"、"怎么和技术委员会解释这个选型"
- 已经在用拼装方案，清楚胶水代码和运维负担，但没时间想替代方案
- 对营销话术免疫。"AI-powered / enterprise-grade / next-generation" 这种词直接跳过
- 只相信：清晰的架构图、具体的数字、读起来像技术白皮书的文案、真实能看到的截图

**Secondary · Agent 使用者 / 开发者** —— 正在用 Claude Code / Cursor / Cline 的开发者。

他们的心智：
- **现在就在用 Agent**，Landing 读到一半可能 Claude Code 窗口还开着
- 不读完长文章就想动手
- 要看到「今天能用」的证据 —— GIF、shell 命令、具体场景
- 相信感官：我试一下就知道

**Landing 必须同时服务这两类人**：CTO 读完全文做决策，开发者读到中段直接复制 shell 命令去试。

### 2. 声音分层 · 两种 voice 的切换规则

| Voice | 用在哪 | 特点 |
|---|---|---|
| **Authoritative**（CTO 视角） | 01 主张 / 02 核心洞察 / 04 能力总览 / 05 硬核数字 / 06 架构内核 | 第三人称陈述，editorial 长段，技术白皮书语调 |
| **Developer voice**（使用者视角） | 03 Claude Code 已能用 / 每个产品页的第 5 段 | 第二人称"你"，当下感，感官验证，每段配可复制 shell |

**Developer voice 五条准则**：

1. **第二人称**："你的 Claude Code"、"你现在就能"、"下次你打开..."
2. **当下感**："30 秒"、"一行命令"、"今天"
3. **具体动作动词**：install / add / 说 / 粘贴 / 复制 —— 不是 "能够 / 支持 / 赋能"
4. **感官验证**："你会看到它还记得"、"第二天你就发现..."、"不再有..."
5. **代码优先**：每段配一行真实可复制的 shell/MCP 命令

两种 voice 的切换点是 **narrative 的中场休息** —— 从紧绷的 authoritative 呼吸一下到 developer，再回到 authoritative 深挖。

### 3. Landing 叙事结构（五屏，锁定屏序）

```
01 · 主张                      Authoritative  + 底部一行 "集成指引" 引导
02 · 工作态 / 学习态             Authoritative  (核心洞察)
──────────────────────────────── ← 声音转折
03 · Claude Code 已能用 ★        Developer voice (zoom in)
──────────────────────────────── ← 声音切回
04 · 四个能力 (可点击卡片)       Authoritative  (zoom out · 链到产品详情)
05 · 硬核数字                    Authoritative  (用户要的效果 · 结尾)
```

**为什么没有「架构内核」屏**：原来第 06 屏是写给 CTO 的技术白皮书式架构图 + BM25/Datalake/加密 compute 的详细叙事。用户明确要求去掉 —— 公共页不该变成技术白皮书，这种内容属于独立的 `/architecture` 白皮书页或文档。Landing 收在硬核数字就够了，让用户看完数字直接去下一步（集成 / 产品详情 / 开始使用）。

**为什么 03 在中段而不是结尾**：读者刚读完 02 的"工作态 vs 学习态"概念时在问"真能用吗？"，03 屏直接回答这个问题。Demo 里展示的正好是 Memory + Knowledge 两个能力，紧接 04 (四个能力) 形成 **zoom in → zoom out** 的镜头语言，读者自然想"我刚看到了其中两个，原来还有两个"。

**每屏内容纲要**：

#### Screen 01 · 主张 (Hero)

- 大字 Source Serif 4 主张，两行：
  - > Agent 的数据层
  - > 不应该是四个产品的拼图。
- Lede（Source Serif 4 Regular, ~22px，暗示对比 + 用户能听懂的效果）：
  - > 过去你要找 4 个组件，自己拼装 Agent 的状态库、知识库、记忆库，还得自己做数据库到数据湖之间的数据同步和加工，把一堆数据工具粘在一起。DBay 让这四件事发生在同一套基础设施上 — 弹性秒级扩缩、完全托管免运维，Agent 可以大胆探索试错，也能从任意断点恢复继续执行。
  - **注意 1**：不要写"Neon"作为拼装组件之一 —— Neon 是 DBay 建立的基础，不是竞品。也不要写"消息队列"。
  - **注意 2**：四件事是 **状态库 · 知识库 · 记忆库 · 数据库到数据湖的同步加工**。不要写"批处理框架跑离线反思"这种描述 —— 传统用户根本没人这么做，那是我们产品化的抽象，不是用户的现实世界语言。
  - **注意 3**：第二句讲 DBay 带给用户的**效果**，不讲技术术语。用户能听懂的是：弹性秒级扩缩、免运维、Agent 探索试错、断点恢复。不要在 hero lede 出现 "copy-on-write / serverless / 多租户 / substrate" 这种词。那些技术术语留到 Screen 06 架构内核屏讲。
- 两个 CTA：`[开始使用]` (暖橙主按钮)、`阅读架构 →` (ghost)
- **底部极小字的 dev shortcut**（开发者双入口）：
  - > 你也可以直接把 MCP 加进 Claude Code:
  - > `$ claude mcp add --scope user dbay -- python -m dbay_mcp`

这行 shell 是给正在读文章、Claude Code 还开着的那个开发者的 —— 他会停下来，复制，去试。

#### Screen 02 · 工作态 / 学习态

- 一张手绘感架构图，左右两列：

**四个产品页的专属主张**（每个都要硬核 + 暗示对比）：

- **Lakebase** — > **PostgreSQL，加 BM25 内核，长在数据湖上。**
  - 三句话三件事：兼容 / 内核改动 / 底座
- **Knowledge** — > **不是 RAG。是 Agent 自己在维护的一本 wiki。**
  - **核心洞察（致敬 Karpathy 最近讲的 LLM wiki 思路）**：LLM-consumable 的知识层形状不是 chunk + embedding，而是**一本会长大的 wiki**。Agent 读 wiki 的条目，不是从几万个 chunk 里捞碎片。
  - **DBay 的做法**：
    - 上传源文档后，DBay 在后台用 LLM 从文档抽出结构化 wiki 页面
    - Claude Code 问问题时**读 wiki 的对应条目**（带原文引用），不是朴素 RAG
    - 用户和 wiki 对话 —— 新问题变成 wiki 的新条目，**知识在增长**
    - 后台持续维护：文档更新时自动调整 wiki，冗余合并，生僻条目降权
  - **双差异化**：
    - 概念层：wiki > RAG（Karpathy 视角）
    - 实现层：四路融合检索（vector + BM25 + 图 + 时序）—— 背后的底层能力，写在第 4 段「DBay 的做法」里作为 HOW，不是主张
  - 第 2 段（场景）开场样例：
    > 你上传项目的 `docs/` 目录。几分钟后，DBay 后台已经把文档拆解、提炼、结构化成一本 wiki。在 Claude Code 里问「我们的 auth flow 怎么处理 refresh token?」—— 它读 wiki 的对应条目回答，带原文引用。
    >
    > 更重要的是：你这一问，wiki 会补上一条新条目「refresh token 处理机制（来自 Jacky 2026-04-11 的问题）」。下次你或者团队成员再问相关问题，wiki 已经有答案了。
    >
    > 这不是一次性的文档检索。这是一本**在生长的**知识库。
  - 第 5 段（今天你就能做）模板：
    > 上传项目文档：
    > `$ dbay knowledge upload ./docs`
    > 
    > 等待 DBay 在后台生成 wiki（几分钟到几十分钟，取决于文档量）。
    > 然后在 Claude Code 里问关于项目的任何问题 —— 它读的是 wiki，不是原始 chunk。
    > 
    > 顺便说一句：接下来你问的新问题会反哺 wiki。知识只增不减。
    > 
    > **（这套思路呼应 Karpathy 最近关于 "LLM consumable knowledge should look like a wiki" 的观点。我们的实现已经跑通，而且你今天就能装上试。）**
  - **Knowledge 的一次检索四路融合**降级为 section 4「DBay 的做法」里的技术细节段落，不作为头版主张。它仍然是重要 moat，但只是实现层面的证据，不是概念层面的 headline。
- **Memory** — > **Agent 的长期记忆，会自己反思你是谁。**
  - 三个硬核 differentiator：加密 / 共享 / **自我反思**
  - "自我反思"是 Memory 的皇冠卖点：每晚 digest 抽 traits，几天后 "反思洞察" 面板里自动出现用户的开发模式、架构偏好、决策风格
  - Memory 产品页第 2 段（场景）**必须**用「三天之后，你打开 Memory 面板的反思洞察标签...」这种具体叙事开场 —— 让读者理解这不是一个 KV 存储，是一个会思考的层
  - 第 5 段（今天你就能做）的 shell 命令后要加一行：> 用几天后回来看「反思洞察」—— DBay 会告诉你它观察到的你。
- **Datalake** — > **多模态算子，DAG 编排，算子间 zero-copy。**
  - 三个具体技术事实，不是"批处理是学习的一部分"的哲学口号

**Memory / Knowledge 产品页第 5 段末尾**要**顺便提 token 节省**（Developer voice 的朋友式 trick），不是硬广告：

- Memory: > 顺便说一句，这套召回比"把历史都塞进 context"省 50-70% 的输入 token。对你的账单和延迟都有可感的差别。
- Knowledge: > 四路融合比朴素向量 RAG 少拿回很多不相关 chunk —— 最终上下文更短，token 成本更低，准确率反而更高。
